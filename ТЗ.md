# **Тестовое задание: сделать мини бекенд \+ фронт для сервиса задач**

### **Контекст**

У нас есть внутренний продукт для управления апартаментами, бронированиями и операционными задачами.


---

# **Задача**

Есть сервис трекер-задач. Нужно доработать флоу:

Если по бронированию возникла проблема, сотрудник создаёт internal task.  
 Задача должна появиться в списке, корректно менять статус, логироваться и не дублироваться.

Предлагается **с нуля написать небольшое веб\-приложение** для демонстрации только этой доработки.

## **1\. Backend API**

Реализовать:

* Сущность `InternalTask`  
* Эндпойнты получения / создания / изменения статуса задач

---

## **2\. Frontend**

Сделать страницу:

/bookings/\[booking\_id\]

На странице блок **Internal Tasks**:

* список задач;  
* форма создания задачи;  
* смена статуса;

UI можно простой. Красота не важна. Важна аккуратная работа.

---

## **3\. Tests**

Минимум покрыть:

* создание задачи;  
* запрет дубля;  
* смену статуса;  
* обработку ошибки API.

Плюсом:

* тест frontend-компонента;  
* тест на валидацию входных данных.

---

## **4\. Logs / Monitoring / Grafana / Sentry**

В `README.md` описать:

* какие события логировать;  
* какие ошибки отправлять в Sentry;  
* какие метрики вывести в Grafana;  
* какой алерт настроить.

Пример метрик в Grafana:

* Internal\_task\_active\_count – Количество активных задач

Пример алерта:

* Если internal\_task\_create\_failed\_total растёт быстрее N ошибок за 10 минут — уведомить Tech Ops.

---

# **Стек**

Обязательно:

* Backend: **Python \+ FastAPI**  
* Frontend: **React \+ Next.js**  
* Tests: pytest или аналог

БД:

* в KeyGo используется **MongoDB**;  
* в тестовом можно использовать любое удобное для того чтобы быстро показать решение

Grafana / Sentry:

* настраивать не нужно;  
* нужно только описать, какие логи, метрики и алерты вы бы добавили.
